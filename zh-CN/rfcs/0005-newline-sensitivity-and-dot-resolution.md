## 1. 摘要

当前结构体字段与枚举变体语法之间没有逗号或分号，靠换行分隔，意味着 Nomo 是**显著换行（newline-sensitive）**语言；但关键字/语法规则只列了关键字，未定义换行/分隔规则，parser 实现缺乏依据。同时 `.` 同时承担模块路径（`std.io`）、变体访问（`Result.Ok`）与字段/方法访问（`self.email`、`items.get`）三种语义，名称解析需要明确消解规则。本 RFC 提出把换行规则与 `.` 消解规则写进规范，倾向于「显著换行作为语句/成员分隔符 + 显式续行规则」「`.` 统一为后缀点访问，由名称解析按左侧实体种类分派」，保持 Draft。

---

## 2. 动机（Motivation）

当前规格所有示例都依赖「一行一字段 / 一行一变体 / 一行一语句」的排版，却没有任何一节定义：

- 换行在什么时候是「分隔符」，什么时候可以续行（如长表达式、`match` 分支）。
- `match` 分支 `Pattern => Expr` 是否也以换行分隔（示例如此但未明文）。
- `.` 在 `std.result.Result`（模块路径 + 类型）、`Result.Ok`（类型 + 变体）、`self.email`（值 + 字段）、`items.get(i)`（值 + 方法）之间如何区分。

没有这些规则，Lexer/Parser无法稳定实现，词法/语法错误码（E0100-E0299）也无法对「缺分隔」「非法续行」给出一致诊断。这直接威胁「Lexer/Parser golden tests 稳定」的验收。

---

## 3. 现状与问题

### 3.1 当前规格现状

结构体字段无分隔符：

```rust
pub struct User {
    pub id: string
    email: string
}
```

枚举变体无分隔符：

```rust
pub enum Color {
    Red
    Green
    Blue
}
```

`match` 分支无分隔符，每行一个 `Pattern => Expr`。`.` 的多重用途散见全文：`std.io`、`std.result.Result`、`Result.Ok(text)`、`self.email`、`items.get(i)`、`fs.read_to_string(path)`。

关键字/语法规则只给出关键字/保留字/字面量，**没有**换行、缩进、分隔符、续行的任何规定。

### 3.2 问题分析

1. **换行语义未定义**：是「换行即分隔」（类 Go/Swift 的自动分号）还是「缩进敏感」（类 Python）？示例看起来是前者，但长表达式、链式调用 `a\n.b()\n.c()`、`match` 多分支的续行规则缺失。
2. **`.` 多义**：
   - `A.B` 中 `A` 可能是模块（`std`）、模块别名（`fs`）、类型/枚举（`Result`）、值/变量（`self`、`items`）。
   - `std.result.Result` 是「模块.模块.类型」；`Result.Ok` 是「枚举.变体」；二者都是点链，靠名称解析区分。
   - 与 [RFC 0007](./0007-unqualified-variant-access.md) 强相关：若引入非限定变体（`Ok` 而非 `Result.Ok`），`.` 的左侧语义判定会变；本 RFC 先把限定形式的消解规则定清。
3. **歧义示例**：`a.b.c` 可能是「模块 a 的模块 b 的成员 c」「值 a 的字段 b 的字段 c」「类型 a... 」。需要分层解析：先按作用域内绑定/导入判定最左标识符的种类，再逐段推进。

---

## 4. 详细设计

### 4.1 换行规则（倾向）

- **显著换行**：换行符是默认的「语句/成员分隔符」。结构体字段、枚举变体、`match` 分支、语句序列均以换行分隔，不使用逗号/分号。
- **显式续行**：当一行在语法上「明显未结束」时自动续行，判定锚点包括：
  - 行尾是二元运算符、`=>`、`,`（参数列表内）、开括号 `(`/`{`/`<` 未闭合。
  - 下一行行首是 `.`（链式调用：`items\n  .get(i)`）。
- **块内排版**：`{ ... }` 内部以换行分隔成员；空行被忽略。
- **诊断**（E0100-E0199 / E0200-E0299）：
  - `E0120` 同一行出现两个成员/语句却无分隔（如把两个字段写在一行）。
  - `E0220` 非法续行 / 期望换行分隔但发现多余 token。
- **备选**：显式分隔符（强制逗号/分号）。更易解析、对工具更友好，但与当前示例排版冲突，需要改写全部示例，且与“克制”气质相比更啰嗦。倾向显著换行以保持现有示例不变。

### 4.2 `.` 消解规则（倾向：统一后缀点 + 名称解析分派）

把 `.` 统一为「后缀点访问表达式」，语法上不区分用途；区分交给**名称解析**按最左标识符的绑定种类分派：

1. 解析 `Head.Seg1.Seg2...` 时，先解析 `Head`：
   - 若 `Head` 是当前作用域内的**值绑定**（变量、`self`、参数）→ 后续 `.Seg` 依次解释为**字段访问**或**方法调用**。
   - 若 `Head` 是**导入的模块/模块别名** → 贪心吃掉连续的「模块」段，直到遇到非模块项（类型/函数/常量），其后转为成员访问。
   - 若 `Head` 是**类型/枚举名** → `.Seg` 解释为**变体**或**关联函数/构造**（如 `Array.new`、`Result.Ok`）。
2. 歧义裁决：值绑定优先级高于同名类型（避免变量被误判为类型）；模块路径只在 `Head` 来自 `import` 时启用。
3. **诊断**（E0300-E0399 名称解析）：
   - `E0320` `.` 左侧标识符未解析（既非绑定也非导入模块/类型）。
   - `E0321` 在值上访问不存在的字段/方法。
   - `E0322` 在枚举上访问不存在的变体。

- **C 后端**：消解后三类点访问分别降为：结构体成员读取（`v.field`）、函数调用（`Pkg_func(...)`，按 12.3 包路径混淆）、变体构造/标签判定（按 4.4 的 `Result_T_E`）。`.` 的多义在 HIR 阶段已消除，C IR 无歧义。

### 4.3 与 [RFC 0007](./0007-unqualified-variant-access.md) 的衔接

本 RFC 固定**限定形式**（`Enum.Variant`）的消解；若 [RFC 0007](./0007-unqualified-variant-access.md) 决定引入 prelude/导入的非限定变体（`Ok`/`None`），名称解析需增加「裸标识符可能是变体」的查找路径——届时由 [RFC 0007](./0007-unqualified-variant-access.md) 在本规则上扩展，不在此处展开。

## 5. 备选方案（Alternatives）

| 维度 | 方案 | 优点 | 缺点 |
| --- | --- | --- | --- |
| 分隔 | 显著换行（倾向） | 保持现有示例、风格克制 | parser 需续行规则、对自动格式化要求高 |
| 分隔 | 显式逗号/分号 | 解析最简单、无歧义 | 改写全部示例、更啰嗦 |
| `.` | 统一后缀点 + 名称解析分派（倾向） | 语法简单、与现有写法一致 | 解析依赖名称解析信息、需定义优先级 |
| `.` | 为模块路径用不同符号（如 `::`） | 词法即可区分模块路径 | 引入新符号、偏离当前统一的 `.` 写法 |

---

## 6. 缺点与风险

- 显著换行 + 续行规则对**自动格式化器**和 AI 生成都提出要求：错误的换行会改变语义，需要在诊断中给出清晰修复建议（贴合 2.2）。
- `.` 统一后缀点意味着 parser 产出的是「未消解的点链」，名称解析阶段才能定型，增加了 AST→HIR 的处理量；但换来语法层简单与示例一致性。
- `::` 方案虽能在词法层分清模块路径，但会让当前全部 `std.io`/`Result.Ok` 写法作废，代价过大。

---

## 7. 对 v0.1 范围的影响

- **建议 v0.1 落地**：把「显著换行 + 续行锚点」与「`.` 统一后缀点 + 名称解析分派」正式写入规范（建议补入关键字/语法规则和编译器架构的名称解析细则）。
- **验收影响**：验收测试矩阵中的 Lexer/Parser golden tests 必须覆盖：一行多成员被拒、链式续行被接受、`A.B.C` 三种消解路径各有样例。
- 不影响交付边界，但**是 Lexer/Parser 能否稳定实现的前置条件**，优先级高。

---

## 8. 倾向性建议（保持 Draft，不拍板）

倾向：**显著换行**作为成员/语句分隔符并定义显式续行锚点；**`.` 统一为后缀点访问**，由名称解析按「值 / 模块 / 类型」三类左侧实体分派，值绑定优先。该方案保持当前全部现有示例不变，同时给 parser 与名称解析提供可实现依据。保持 Draft。

---

## 9. 未决问题

- 续行锚点的完整清单（是否允许行首二元运算符续行，如下一行以 `+` 开头）。
- `match` 分支体为块 `{ ... }` 时的换行规则与单表达式分支是否统一。
- 与 [RFC 0007](./0007-unqualified-variant-access.md) 引入非限定变体后的名称解析优先级冲突如何排序。
