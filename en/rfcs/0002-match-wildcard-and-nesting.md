## 1. Summary

The pattern-matching specification keeps `_` disabled so exhaustiveness diagnostics enumerate every missing arm. Nested `Option`/`Result` boilerplate is addressed with `let else`, `if let`, and the existing postfix `?`. This RFC accepts the A + C combination, which is implemented across the parser, type checker, C backend, and example tests.

---

## 2. Motivation

Exhaustiveness is Nomo's safety selling point: all arms must be explicitly covered, and diagnostics report missing arms one by one. But when "disabling `_`" is combined with "values are often nested", it forces out deeply-nested pyramid code. The `swap` in the array-swap example is merely "take two elements, swap only if both exist", yet it is written with a double `match` and two repeated `panic("index out of bounds")`. This kind of code:

- Is hard to write, hard to read, error-prone (the two `None` arms must each be written once).
- Is prone to missing arms or wrong nesting levels when generated by AI.
- Conflicts with the 2.2 goal of "reducing generation and fixing cost".

---

## 3. Status and Problem

### 3.1 Current Specification Status

The current pattern-matching specification:

> `match` must be exhaustive over all arms. v0.1 does not yet support the `_` wildcard arm, prioritizing diagnostics that cover all missing arms.

The array-swap example reflects the nesting cost:

```rust
fn swap(mut items: Array<i32>, i: u64, j: u64) {
    let a = items.get(i)
    let b = items.get(j)

    match a {
        Option.Some(av) => {
            match b {
                Option.Some(bv) => {
                    items.set(i, bv)
                    items.set(j, av)
                }
                Option.None => panic("index out of bounds")
            }
        }
        Option.None => panic("index out of bounds")
    }
}
```

### 3.2 Problem Analysis

- **Repetition**: `Option.None => panic(...)` appears twice; logically "panic on any out-of-bounds", yet it is forced to be scattered across two levels.
- **Nesting depth grows linearly with the number of matched values**: two `Option`s are already two levels, three values is three levels.
- **The double-edged nature of `_`**: `_` can eliminate verbosity, but it also swallows the "an added variant is unhandled" diagnostic — which is exactly what 3.8 wants to avoid. Opening up `_` directly would sacrifice the exhaustiveness dividend.

The core tension: **exhaustiveness diagnostics (want everything listed)** and **nested ergonomics (want to omit)** are hard to have both; a balance point must be found between an "omission mechanism" and "exhaustiveness protection".

---

## 4. Detailed Design

### 4.1 Option A: Keep `_` disabled (status quo)

- **Semantics**: unchanged.
- **Advantages**: strongest exhaustiveness diagnostics, simplest semantics.
- **Disadvantages**: no solution for nesting boilerplate.

### 4.2 Option B: Conditionally allow `_`

- **Syntax**: allow `_ => ...`, but only legal when the number of remaining variants it covers is ≥ 2; if `_` actually covers only 1 remaining variant, the compiler hints "use the named variant for clarity".
- **Semantics**: no arm may follow `_`; introduce a lint (warning level) for "`_` masked an added variant", so that adding a variant to the enum still triggers a reminder.
- **Diagnostics**: add a name-resolution/type-checking warning, e.g. `E0410` (`_` masked enumerable remaining variants).
- **Disadvantages**: once opened, AI/users will tend toward "always `_`", gradually eroding the exhaustiveness culture.

### 4.3 Option C: Introduce `if let` / `let else` (accepted)

- **Syntax**:

```rust
fn swap(mut items: Array<i32>, i: u64, j: u64) {
    let Option.Some(av) = items.get(i) else { panic("index out of bounds") }
    let Option.Some(bv) = items.get(j) else { panic("index out of bounds") }
    items.set(i, bv)
    items.set(j, av)
}
```

- **Semantics**: `let PATTERN = expr else { ... }`, where a match failure goes to `else` (`else` must diverge: `panic`/`return`/`break`). `if let PATTERN = expr { ... }` is used when "only one branch is of interest".
- **C backend**: expands to a single-arm test + failure branch, equivalent to a subset of `match`, with no new runtime.
- **Diagnostics**: keep `match` exhaustiveness unchanged (`match` still disables `_`); `if let`/`let else` are semantically "non-exhaustive extraction" by nature and do not affect `match`'s full-coverage dividend.
- **Advantages**: without opening up `_`, it eliminates "single-branch extraction + early exit", the most common source of nesting, and the divergent `else` does not silently swallow variants.

### 4.4 Option D: `?`-style Option destructuring

- **Syntax**: in a function that returns `Option`, allow `let av = items.get(i)?`, returning `None` early when `None`.
- **Semantics**: same source as the `?` of [RFC 0001](./0001-error-propagation-and-conversion.md), but acting on `Option`, requiring the current function to return `Option`.
- **Disadvantages**: `swap` returns `void`/will `panic`, not `Option`, so it does not directly apply to this example; its applicability is narrower than C.

---

## 5. Alternatives

| Option | Approach | Advantages | Disadvantages |
| --- | --- | --- | --- |
| A Keep disabled | Unchanged | Strongest exhaustiveness, simplest | No solution for nesting boilerplate |
| B Conditional `_` | Restricted wildcard + lint | Directly reduces burden | Erodes exhaustiveness culture, needs an extra lint |
| C `if let`/`let else` (accepted) | Single-branch extraction + divergent else | Preserves `match` exhaustiveness, removes the main nesting source | Adds syntactic surface area |
| D Option `?` | Early exit on `Option` | Consistent with `?` | Only applies to functions returning `Option` |

---

## 6. Drawbacks and Risks

- Option C adds a syntactic point; the parsing of `let ... else` must be clarified in the keyword/grammar rules (the `else` block must diverge), and distinguished from the `if`-expression semantics.
- If both C and D are introduced, consistency with the `?` of [RFC 0001](./0001-error-propagation-and-conversion.md) must be coordinated to avoid three overlapping "early exit" forms, which would violate 2.2.
- Option B's "remaining variant counting" depends on the name-resolution stage already knowing all variants of the enum; in implementation it shares the same information as exhaustiveness checking, so the cost is controllable but the semantic boundary must be written clearly.

---

## 7. Impact on v0.1 Scope

- **Landed in v0.1**: Option C's diverging `let else` and `if let`; Option D is served by the existing postfix `expr?` model for functions returning `Option`.
- **Unchanged**: `match` continues to disable `_` (Option A), locked by parser and exhaustiveness tests.
- **Defer**: Option B (restricted `_`) is left for v0.2 discussion, to avoid weakening exhaustiveness culture in v0.1.
- **Acceptance impact**: the acceptance test matrix's Parser/Type checker tests need to add success/failure cases for `let else` (a non-diverging `else` should error).

---

## 8. Decision

Accept **the A + C combination**: `match` keeps `_` disabled, while diverging `let else` and `if let` absorb the main boilerplate source of nested `Option`/`Result`. A non-diverging `else` is rejected by type checking, and both constructs are covered by the current implementation and acceptance tests.

---

## 9. Follow-up Questions

- If `_` is still to be introduced in the future, is it "globally opened" or "only top-level `match`"?
- Future richer nested destructuring should continue to reuse the current exhaustiveness and binding-scope rules.
