## 1. 摘要

当前内存模型规格 对 `string` 和 `Array<T>` 都规定「值语义 + 引用计数（ARC）+ 写时复制（COW）」，并叠加 `mut` 借用唯一性。这是 v0.1 运行时中实现复杂度最高、最易超期的部分：需要正确处理引用计数增减、COW 触发判定、与 C 后端值传递语义的交互，以及并发尚未存在时「ARC 是否需要原子」的取舍。本 RFC 评估实现成本，并讨论 v0.1 是否可退化为更简单策略。倾向于「v0.1 用非原子引用计数 + COW，但把 COW 限定在少数明确写操作上，并允许把 `string` 先实现为不可变共享（无 COW，因其本就不可变）」，保持 Draft。

---

## 2. 动机（Motivation）

交付边界把 `std.array`、`std.string` 列为必交付，当前性能说明承诺「字符串和动态数组在标准库中实现引用计数与写时复制」。这意味着 ARC+COW 不是可选项，而是 v0.1 验收内容。但它牵涉：

- 运行时函数（retain/release/clone-on-write）的正确性与内存安全。
- COW 触发时机判定（何时 `refcount > 1` 且即将写）。
- 与 C99 值语义、与 `mut` 借用、与 `defer`清理顺序的交互。

如果低估这块工作量，最可能拖垮 v0.1 的「链路完整」目标。

---

## 3. 现状与问题

### 3.1 当前规格现状

- 内存分类：`string`、`Array<T>` 归为「标准库托管值：引用计数 + 写时复制」。
- 字符串：`string` **不可变**，赋值增加引用计数，拼接生成新串，C 后端用运行时函数管理计数。
- 动态数组：`Array<T>` 读共享底层存储，写在「引用计数 > 1」时触发 COW，通过 `mut` 授权修改。
- 性能承诺：性能「以基准测试为准」，未承诺接近 C/Rust。
- C 后端：C 后端要「生成可读 C」「标准库运行时以 C 源文件链接」。

### 3.2 问题分析

1. **`string` 其实不需要 COW**：当前字符串设计说明字符串不可变、拼接生成新串。既然不可变，就永远不会「就地写」，COW 触发点不存在——`string` 只需共享 + 引用计数，COW 是 `Array<T>` 才真正需要的机制。当前规格把两者并列为「ARC+COW」属于过度表述。
2. **COW 触发判定要可靠的「唯一性」信息**：`Array.push`/`set`在 `refcount > 1` 时复制。判定依赖运行期 `refcount`，而 `mut` 借用唯一性（见 [RFC 0004](./0004-mutable-borrow-uniqueness.md)）是编译期检查。两套唯一性语义（编译期借用唯一 vs 运行期计数唯一）需要协调，否则会出现「编译期认为唯一、运行期计数 > 1」的歧义。
3. **原子性**：v0.1 无并发（`go`/`chan` 已推迟）。此时引用计数无需原子操作。但若运行时一开始就写成原子（为将来铺路），会平白增加单线程开销；若写成非原子，将来并发时要重写。
4. **与 C 后端、`defer` 交互**：值离开作用域要 `release`；`?` 早退、`defer` 逆序执行都需在 codegen 中正确插入 release 点，遗漏即内存泄漏或 use-after-free。

---

## 4. 详细设计

### 4.1 方案 A：完整 ARC + COW（当前规格字面）

- **运行时**：`nomo_rc_retain` / `nomo_rc_release` / `nomo_cow_make_unique`，`Array<T>` 头部含 `refcount + len + cap + data`。
- **C 后端**：每个绑定移动/复制点插入 retain/release；写操作前调用 `make_unique`。
- **成本**：最高。需完整覆盖移动语义、提前返回、`defer` 顺序，测试矩阵大。
- **风险**：最可能成为 v0.1 长尾。

### 4.2 方案 B：分而治之（倾向）

- **`string`**：实现为**不可变共享 + 引用计数，无 COW**。因不可变，永不就地写，省掉 COW 全部复杂度；拼接（8.5 `concat`）始终分配新串。
- **`Array<T>`**：保留 **引用计数 + COW**，但把 COW 触发面收敛到明确的写 API（`push`/`set`），写前 `make_unique`。
- **引用计数**：v0.1 用**非原子**计数（无并发，见 1.2），在运行时头部注释「并发版本见 v0.3 RFC」。
- **诊断/codegen**：release 点插入与 `defer`/`?` 早退统一在 HIR→C IR 阶段处理（12.2 三层 IR）。
- **优点**：把不必要的 `string` COW 砍掉，集中精力把 `Array<T>` COW 做对，工期可控且不违背 5.x 的对外语义（用户看到的仍是值语义）。

### 4.3 方案 C：v0.1 退化为纯拷贝（无引用计数）

- **做法**：`string`/`Array<T>` 赋值即深拷贝，完全不引用计数、不 COW。
- **优点**：实现最简单、最易测、绝不泄漏。
- **缺点**：违背性能与内存模型的明文承诺；大数组频繁拷贝性能差；将来切回 ARC 是行为可见变化（虽语义等价于值语义，但性能特征不同）。可作为「应急回退路线」。

### 4.4 方案 D：纯引用计数、无 COW（共享可变？）

- 与值语义冲突：无 COW 的共享可变会让「值语义」破功（一处改、另一处可见）。除非禁止 `Array` 写后别名，否则不可取。故仅适用于不可变类型（即等价于对 `string` 的方案 B 处理）。

---

## 5. 备选方案（Alternatives）

| 方案 | 做法 | 优点 | 缺点 |
| --- | --- | --- | --- |
| A 完整 ARC+COW | 字面实现两类型 | 完全符合当前内存模型 | 成本最高、最易超期 |
| B 分而治之（倾向） | string 无 COW、Array 有 COW、非原子 | 砍掉伪需求、聚焦真复杂点、语义不变 | 仍需把 Array COW 做对 |
| C 纯拷贝退化 | 全深拷贝 | 最简单最安全 | 违背性能与内存模型承诺、性能差 |
| D 纯 RC 无 COW | 共享可变 | 实现简单 | 破坏值语义，不可用于可变 Array |

---

## 6. 缺点与风险

- 方案 B 仍要实现 `Array<T>` 的 COW 与 release 点插入，这是真正的难点；需要充分的 Mutability/Codegen 测试。
- 非原子计数是一次性技术债：v0.3 引入并发时必须切原子或换策略，需在运行时源码与 RFC 中显式标注，避免被误用于跨线程。
- 编译期借用唯一性（[RFC 0004](./0004-mutable-borrow-uniqueness.md)）若足够强，理论上某些写可以「就地改而不查 refcount」；但 v0.1 借用检查若较弱（见 [RFC 0004](./0004-mutable-borrow-uniqueness.md) 倾向），仍需运行期 refcount 兜底，二者关系要在两篇 RFC 间锁定。

---

## 7. 对 v0.1 范围的影响

- **建议 v0.1 落地**：方案 B。`string` 走「不可变共享 + 非原子 RC」；`Array<T>` 走「非原子 RC + COW（仅 `push`/`set` 等写 API 触发）」。
- **建议当前规格修订**：把当前字符串设计中的「ARC+COW」更正为「`string` 仅引用计数共享（不可变，无需 COW）」，避免误导实现者。
- **保留回退闸门**：若 `Array<T>` COW 在发布前仍不稳定，允许临时切换到方案 C（纯拷贝）保住「链路完整」，COW 作为紧随其后的补丁。
- **验收影响**：验收测试矩阵需新增「COW 触发（refcount>1 写）」「release 点（`defer`/`?` 早退）」的运行时 smoke + 内存检查（如 ASan）。

---

## 8. 倾向性建议（保持 Draft，不拍板）

倾向 **方案 B**：区分 `string`（不可变，免 COW）与 `Array<T>`（RC+COW），统一用非原子计数，并把纯拷贝（方案 C）作为发布前的应急回退。这样在不削弱对外值语义的前提下，显著降低 v0.1 运行时的实现与超期风险。保持 Draft。
